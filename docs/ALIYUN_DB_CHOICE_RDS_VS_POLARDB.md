# 阿里云数据库选型：RDS 够不够？MySQL 还是 PostgreSQL？

针对当前博客场景的简要结论与理由。

---

## 一、RDS 够不够？要不要上 PolarDB？

**结论：博客场景用 RDS 就足够，不必上 PolarDB。**

| 对比项 | RDS | PolarDB（云原生） |
|--------|-----|-------------------|
| 定位 | 通用关系型数据库，单机/主从 | 云原生、存算分离、一写多读扩展 |
| 适用场景 | 中小型应用、博客、后台、常规 Web | 高并发读、大库、读写分离、高可用要求极高 |
| 价格 | 相对便宜，按规格/包月 | 相对更贵，按规格 + 存储等 |
| 博客场景 | ✅ 完全够用 | ❌ 性能与能力过剩，成本不划算 |

**博客典型特征**：读多写少、QPS 不高、数据量小、单库单实例即可。RDS 的 PostgreSQL/MySQL 主实例（甚至基础规格）就能满足，无需 PolarDB 的读写分离、弹性扩展等能力。

**建议**：直接选 **阿里云 RDS**，规格选 1 核 1G 或 1 核 2G 的入门款即可起步，后续流量上来再升配。

---

## 二、选 MySQL 还是 PostgreSQL？

**结论：选 PostgreSQL，且你当前项目已经是 PostgreSQL，无需改成 MySQL。**

### 1. 和当前项目一致（最重要）

- 你现在的 Prisma 配置是 `provider = "postgresql"`，模型、迁移、查询都是按 PostgreSQL 来的。
- 若改用 MySQL，需要：
  - 改 `provider` 为 `mysql`
  - 检查并调整类型/语法差异（如布尔、JSON、字符串长度、自增等）
  - 重新做迁移或导数据
- 继续用 PostgreSQL，可以直接把 `PRISMA_DATABASE_URL` 指到阿里云 RDS PostgreSQL，**零迁移、零改代码**。

所以从“少折腾、少风险”角度，**选 PostgreSQL 最合适**。

### 2. 若从零选型，MySQL vs PostgreSQL（博客场景）

| 维度 | MySQL | PostgreSQL |
|------|--------|-------------|
| 生态/运维 | 非常常见，文档多、运维熟手多 | 也很常见，云厂商都支持 |
| 与 Prisma | 支持良好 | 支持良好，很多 Prisma 示例和文档用 PostgreSQL |
| JSON 支持 | 有 JSON 类型，功能够用 | JSON/JSONB 很强，适合以后加元数据、配置等 |
| 全文检索 | 有基础能力 | 全文检索、扩展更丰富，若博客要做站内搜索更顺手 |
| 许可与生态 | GPL / 商业双轨 | PostgreSQL 协议，更宽松 |
| 阿里云 RDS | ✅ 支持 | ✅ 支持，规格从低到高都有 |

对博客这种读多写少、表结构不复杂、可能有一点 JSON 或全文搜索的场景，**PostgreSQL 在功能上更顺手，且和你现有技术栈一致**。

---

## 三、直接可执行的选型

1. **产品**：阿里云 **RDS**（不必选 PolarDB）。
2. **引擎**：**PostgreSQL**（与当前 Prisma 一致，无需改代码或迁库）。
3. **规格**：入门可选 **1 核 1GB** 或 **1 核 2GB**，存储按需（如 20GB 起），后续再升配。
4. **连接方式**：在 RDS 控制台创建实例、白名单、账号后，把连接串填到 `PRISMA_DATABASE_URL`（或 `DATABASE_URL`），在服务器上执行 `prisma migrate deploy` 或 `prisma db push` 即可。

这样在成本可控的前提下，和现有博客架构完全兼容，无需为 PolarDB 或换成 MySQL 多花钱、多改代码。
